✅ مُصلح console.error في catch block recitations.tsx : line 142 منخفضة
تم وجود console.error(e) في catch block الخاص بـ playJuz. يؤدي إلى ظهور رسائل غير ضرورية في developer console.
✅ الإصلاح: استُبدل بـ comment توضيحي. الحالة الآن: silent error handled by UI state.
⚠️ تحذير عدم معالجة أخطاء الشبكة في wird.tsx wird.tsx متوسطة
شاشة الورد اليومي تعمل بشكل كامل offline (بيانات ثابتة)، لذا لا تتأثر بأخطاء الشبكة. لكن لا توجد معالجة لأخطاء AsyncStorage المحتملة عند تحميل التقدم اليومي.
💡 الحل المقترح: أضف try/catch في دالة loadTodayProgress:

const data = await AsyncStorage.getItem(key);
→ لف بـ try/catch وأضف fallback: setProgress({})
⚠️ تحذير أيقونات expo-symbols قد لا تعمل على Android _layout.tsx + جميع الشاشات متوسطة
expo-symbols تعمل بشكل مثالي على iOS (SF Symbols). على Android والويب، يعتمد IconSymbol على expo-vector-icons كـ fallback. بعض الأيقونات كـ scroll.fill و text.book.closed.fill قد تظهر بأيقونة بديلة.
💡 الحل: مراجعة ملف components/ui/icon-symbol.tsx والتأكد من وجود mappings لكل الأيقونات المستخدمة في المشروع.
⚠️ تحذير ViewShot قد يتطلب إذن تخزين على Android 13+ favorites.tsx : handleExportImage متوسطة
على Android 13+ (API 33)، الإذن WRITE_EXTERNAL_STORAGE مُهمَل. يجب استخدام MediaLibrary.requestPermissionsAsync() بدلاً منه — وهذا موجود بالفعل في الكود. لكن في بعض الأجهزة قد يفشل captureRef قبل رفع الإذن.
✅ الكود الموجود يطلب الإذن صح. تأكد من أن app.json يحتوي على: "permissions": ["READ_MEDIA_IMAGES"] في قسم Android.
ℹ️ ملاحظة Network timeout غير محدد lib/quran-api.ts + lib/prayer-api.ts منخفضة
جميع fetch calls لا تحدد timeout. في حالة ضعف الاتصال، المستخدم ينتظر إلى ما لا نهاية.
💡 الحل المقترح: استخدم AbortController مع timeout:
const controller = new AbortController(); const timeout = setTimeout(() => controller.abort(), 10000); const res = await fetch(url, { signal: controller.signal }); clearTimeout(timeout);
ℹ️ ملاحظة quran-search عند truncated content quran-search.tsx : آخر سطر منخفضة
الملف ينتهي بـ View style={[s.e... — يبدو أنه مُبتور في عرض الكود، لكن الملف الأصلي كامل (702 سطر، 30KB). هذا فقط في العرض المقتطع وليس خطأ حقيقي.

🤖 النشر على Google Play
▶️
Google Play Store
Android — رسوم التسجيل: 25$ مرة واحدة فقط
⏱️
المدة الإجمالية: 3-7 أيام (من البداية حتى ظهور التطبيق في المتجر)

المرحلة 1: الإعداد البيئي (يوم 1)

  1. تثبيت EAS CLI
    أداة Expo Application Services لبناء التطبيقات
    npm install -g eas-cli eas login
  2. تهيئة EAS في المشروع
    داخل مجلد المشروع
    cd improved/ eas build:configure
  3. تحديث app.json
    تأكد من صحة هذه الحقول في app.json
    { "expo": { "name": "القرآن الكريم", "slug": "quran-app", "version": "1.0.0", "android": { "package": "com.yourname.quranapp", // اسم فريد "versionCode": 1, "permissions": [ "ACCESS_FINE_LOCATION", "ACCESS_COARSE_LOCATION", "POST_NOTIFICATIONS", "READ_MEDIA_IMAGES" ] } } }
  4. إنشاء Keystore (مفتاح التوقيع)
    ⚠️ احتفظ بهذا الملف بأمان — ستحتاجه في كل تحديث مستقبلي!
    # EAS سينشئه تلقائياً ويحفظه بأمان eas credentials

المرحلة 2: بناء التطبيق (يوم 1-2)

  1. بناء APK للاختبار المحلي
    للتجربة على جهازك قبل الرفع
    eas build --platform android --profile preview # يستغرق 10-20 دقيقة على خوادم EAS
  2. بناء AAB للنشر الرسمي
    Google Play يفضل AAB على APK
    eas build --platform android --profile production # ينتج ملف .aab جاهز للرفع
  3. تحميل الملف المبني
    بعد اكتمال البناء، حمّل ملف .aab من رابط EAS
    eas build:list # ثم اضغط على الرابط لتحميل الملف

المرحلة 3: إعداد حساب Google Play (يوم 2)

  1. فتح حساب Google Play Console
    اذهب إلى play.google.com/console → ادفع 25$ → أنشئ تطبيق جديد
  2. إعداد بيانات التطبيق
    ستحتاج لإعداد:
    • اسم التطبيق (بالعربية والإنجليزية)
    • وصف قصير (80 حرف) ووصف كامل (4000 حرف)
    • أيقونة 512×512px (PNG)
    • صور Feature Graphic: 1024×500px
    • لقطات شاشة: 4-8 صور (هاتف + تابلت اختياري)
    • تصنيف العمر والمحتوى
    • سياسة الخصوصية (URL لازم)
  3. رفع ملف AAB
    Release → Production → Create new release → ارفع ملف .aab
  4. مراجعة ونشر
    مراجعة كل المعلومات → Submit for review → انتظر 3-7 أيام
💡
نصيحة مهمة: ابدأ بـ "Internal Testing" لـ 5-10 مختبرين قبل النشر العام — يوفر وقت المراجعة ويكشف المشاكل مبكراً.

# نشر تحديث Over The Air (بدون بناء جديد) eas update --branch production --message "إصلاح أخطاء" # زيادة رقم الإصدار # في app.json: "versionCode": 2 (زد بمقدار 1 في كل إصدار) # بناء للإصدار الجديد eas build --platform android --profile production --auto-submit

Apple App Store

⚠️
مهم جداً: بناء iOS يتطلب جهاز Mac أو استخدام EAS Build على السحابة (لا يحتاج Mac). لكن الرفع على App Store Connect يمكن عمله من أي جهاز.

المرحلة 1: إعداد Apple Developer (يوم 1)

  1. سجّل في Apple Developer Program
    اذهب إلى developer.apple.com → Enroll → ادفع 99$/سنة → انتظر 24-48 ساعة للموافقة
  2. تحديث app.json لـ iOS
    أضف Bundle ID الخاص بك
    { "expo": { "ios": { "bundleIdentifier": "com.yourname.quranapp", "buildNumber": "1", "infoPlist": { "NSLocationWhenInUseUsageDescription": "لتحديد اتجاه القبلة وأوقات الصلاة", "NSLocationAlwaysUsageDescription": "لإشعارات أوقات الصلاة" }, "supportsTablet": true } } }
  3. ربط EAS بحساب Apple
    EAS سيُنشئ Certificates و Provisioning Profiles تلقائياً
    eas credentials --platform ios # سيطلب Apple ID وكلمة المرور لمرة واحدة

المرحلة 2: بناء IPA (يوم 2)

  1. بناء للمحاكي (للاختبار)
    للتجربة في iOS Simulator
    eas build --platform ios --profile preview
  2. بناء للنشر الرسمي
    ينتج ملف .ipa جاهز للرفع
    eas build --platform ios --profile production # يستغرق 15-30 دقيقة

المرحلة 3: رفع على App Store Connect (يوم 2-3)

  1. إنشاء التطبيق في App Store Connect
    appstoreconnect.apple.com → My Apps → + New App
  2. رفع الـ IPA
    يمكن عبر EAS مباشرة أو عبر Transporter app
    # رفع تلقائي عبر EAS eas submit --platform ios --latest # أو تحميل IPA وترفعه يدوياً عبر Transporter
  3. إعداد صفحة التطبيق
    ستحتاج:
    • أيقونة 1024×1024px (بدون زوايا مدورة — Apple تضيفها)
    • لقطات شاشة iPhone 6.9" (3-10 صور)
    • لقطات iPad 12.9" (اختياري لكن يُحسّن التقييم)
    • وصف التطبيق (4000 حرف)
    • Keywords (100 حرف) — مهمة للـ ASO
    • Support URL و Marketing URL
    • سياسة الخصوصية (Privacy Policy URL)
  4. تقديم للمراجعة
    Submit for Review → انتظر 24-72 ساعة
🚨
أسباب الرفض الشائعة من Apple:
  • غياب Privacy Policy URL
  • أيقونة تحتوي نصاً بلغات متعددة
  • Permission descriptions غير واضحة في Info.plist
  • التطبيق يتعطل أثناء المراجعة (اختبر جيداً!)
